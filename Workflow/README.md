# AI Frontier 新闻编辑工作流（Schema v2）

应用只展示已经完成采集、中文改写和独立审稿的成品资讯。采集规则、可信度判断和审稿过程留在后台，不进入读者正文。

## 两个独立维度

- `contentType` 表示内容形态：`论文`、`开源项目`、`模型发布`、`Agent产品`、`产业事件`。
- `topics` 表示内容主题：`大模型`、`Agent`、`重要研究`、`开源项目`、`产业动态`，一条资讯可以有多个主题。

例如，研究 Agent 评测方法的 arXiv 文章仍是 `论文`，同时带有 `Agent` 和 `重要研究` 主题；Langflow 的正式版本仍是 `开源项目`，同时带有 `Agent` 主题。报道结构由 `contentType` 决定，栏目和推荐由 `topics` 决定。

## 发布流水线

1. 从官方博客、论文索引、正式 GitHub Release 和专业技术媒体采集候选事件。
2. 在撰稿前删除娱乐、人物花边、纯融资、纯营销和普通代码推送。
3. 使用 DOI、arXiv 编号、仓库版本和近似标题合并同一事件。
4. 第一阶段 AI 只依据原始材料写中文报道：摘要结论先行，正文动态组织为 3–5 段。
5. 第二阶段 AI 独立比较原始材料和成稿，检查事实支持、分类、中文完整性、重复和元话语。
6. 程序再次执行确定性门槛；只有审稿通过且技术相关性、创新性合格的条目才能进入当期数据。
7. 当合格条目不足 20 条、不是 10 的整倍数，或任一批缺少大模型、Agent、论文、开源项目时，不覆盖上一期成品。

这是原子发布：单条失败即整条丢弃，整期失败即保留上一期，不会把英文 RSS、README 或半成品显示给读者。

## 报道要求

- 标题、摘要和首段首先回答“做到了什么、贡献或关键变化是什么”。
- 摘要为 70–140 个中文字符；正文总计至少 275 个中文字符。
- 段落标题跟随事件内容，不固定套用五段模板。
- 论文优先交代问题、方法、实验结果和贡献；项目优先交代能力、机制、上手方式和成熟度；模型与 Agent 发布优先交代新增能力、效果、实现路径和具体限制。
- 产品名、模型名和必要术语可以保留英文，完整英文句子不得发布。
- 筛选逻辑、置信度、来源声明、核查提醒和编辑流程不得写入读者内容。

## GitHub 项目规则

GitHub Search 仅用于发现候选仓库。程序随后读取 `releases/latest`，只有处于时间窗内、包含实质发行说明的正式 Release 才能进入撰稿环节。Star 数和普通 push 不构成新闻事件。

## 运行

```powershell
$env:AI_API_KEY = "..."
$env:AI_API_BASE = "https://models.github.ai/inference"
$env:AI_MODEL = "openai/gpt-4.1"
$env:AI_REVIEW_MODEL = "openai/gpt-4.1"
$env:GITHUB_TOKEN = "..."
python scripts/update_feed.py --config Data/source-feeds.json --output Data/news.json --drop-existing
```

`AI_REVIEW_MODEL` 可与撰稿模型不同。缺少 AI 编辑服务时，脚本不会发布英文回退内容，只保留已有的合格中文版。
